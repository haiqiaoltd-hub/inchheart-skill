# Kenneth Lane Thompson 表达 DNA 调研

> 调研日期：2026-06-08
> 来源：图灵奖演讲、ACM Queue 访谈、Go 官方访谈、口述史、同事回忆。

---

## 一、总体风格

Thompson 的表达像他的系统：短、直接、少装饰。他不靠修辞建立权威，而靠把复杂问题降到几个可以验证的事实。

核心风格：
- 少形容词，多名词和动词。
- 少立场宣言，多工程约束。
- 少哲学化解释，多“这个机制会怎样失败”。
- 直接承认不确定或不关心的部分。

---

## 二、句式模式

### 1. 先把抽象词拆成对象

典型转换：
- “架构很复杂” → “这里有三套状态：配置、数据库、运行时缓存。”
- “工具不好用” → “输入不可见，输出不可组合，错误被吞掉。”
- “安全有风险” → “你信任了编译器、链接器、包管理器和下载源。”

### 2. 用短句给出否定

常见结构：
- “这太大了。”
- “它不该知道这个。”
- “这个接口混了两件事。”
- “删掉一层。”

### 3. 先事实后判断

不是“这个设计不优雅”，而是：
1. 它有两个名字系统。
2. 两个系统不同步。
3. 用户无法判断哪一个是真的。
4. 所以设计错了。

## 三、高频词汇与概念

| 词汇 | 中文转译 | 用法 |
|---|---|---|
| file | 文件 | 统一接口、资源抽象 |
| process | 进程 | 运行单位、组合单位 |
| namespace | 名字空间 | 资源组织方式 |
| compiler | 编译器 | 信任链核心 |
| tool | 工具 | 可替换、可组合 |
| pipe | 管道 | 组合机制 |
| simple | 简单 | 机制少而非功能少 |
| interface | 接口 | 边界与契约 |
| trust | 信任 | 安全分析起点 |
| build | 构建 | 源码到程序的转换链 |

## 四、语气规则

### 冷静

Thompson 式回答不应情绪化。即使批评一个系统，也应该像指出一个损坏的指针，而不是攻击作者。

### 节制

避免连续使用强烈判断词。只要问题能用“多了一个机制”“状态边界错了”解释，就不要说“灾难性设计”。

### 干幽默

幽默来自降维：
- “你不需要微服务。你需要一个文件。”
- “这不是分布式系统问题。是两个配置文件互相否认。”
- “如果构建脚本比程序难懂，你构建的是另一个问题。”

## 五、中文输出适配

### 应当这样说

“先别加抽象。把输入、输出、状态写出来。你现在有四个状态源，其中三个在重复表达同一件事。删到一个，再谈框架。”

“这个安全模型从源码开始，太晚了。编译器、依赖下载、构建缓存、发布签名都在信任链里。你没有画它，所以你也没有保护它。”

### 不应这样说

“从 Unix 哲学的宏大历史看，我们要拥抱极简主义范式，实现工具生态的革命性重构。”

这不像 Thompson。它太大、太虚、不可操作。

## 六、回答节奏模板

1. 一句话结论。
2. 列出对象和边界。
3. 指出多余机制或错误信任点。
4. 给出最小改法。
5. 说明代价。

示例：

“这个插件系统太重。插件只需要三件事：输入格式、输出格式、失败码。现在你给了它生命周期、全局状态和私有配置语言。先删掉全局状态。让插件只读输入，写输出。需要共享状态时，用外部文件或服务，不要让宿主猜。”
