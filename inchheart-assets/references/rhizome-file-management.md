# 块茎式索引与文件管理

> 本附录是 `inchheart-assets` 的补充视角。主文档负责可执行的文件归位标准；本附录只解释块茎/图谱为什么适合做关联索引，不适合做物理分类。

## 先纠正概念

不要说“块茎分类”。

更准确的说法：

- 块茎式索引
- 图谱式关联
- 非树状语义导航
- 多入口知识地图

原因很简单：分类要回答“这个东西放哪”，而块茎回答“这个东西还能通向哪里”。

| 问题 | 分类系统 | 块茎/图谱 |
|---|---|---|
| 是否要求唯一位置 | 是 | 否 |
| 是否允许多重归属 | 有限 | 天然允许 |
| 主要数据结构 | 树 | 图 |
| 适合对象 | 文件夹、媒体库、备份结构 | 笔记、灵感、主题关系 |
| 失败方式 | 分类冲突、目录过深 | 连接稀疏、命名混乱 |

一句话：

> 文件夹是归位系统；块茎是抵达系统。

## 为什么单一树状分类会失效

树状分类本身并不低级。它的问题是：任何一棵树只能服务一个主轴。

一本《AI 与哲学》可以按很多标准成立：

| 主轴 | 归位 |
|---|---|
| 学科 | 哲学或计算机 |
| 用途 | 某个研究项目 |
| 状态 | 待读、在读、已读 |
| 作者 | 作者目录 |
| 格式 | PDF、EPUB |
| 来源 | 购买、下载、扫描 |

这些答案不一定谁对谁错。真正的错误是把它们放进同一层文件夹里。

坏结构：

```
EbookLibrary/
├── 哲学/
├── AI/
├── 已读/
├── 2026/
└── 项目资料/
```

这不是“树状不行”，而是没有选择唯一主轴。

对书库还要再加一层判断：不同学科的树状主轴不同。哲学可以按思想传统和历史谱系，历史应按地域与时间，技术应按功能与工具链。不要把哲学史分类法套给所有书。

## 块茎适合解决什么

块茎适合解决“多重入口”和“跨领域联想”。

例子：

```markdown
# AI 与哲学

关联：
- [[技术哲学]]
- [[生成式 AI]]
- [[主体性]]
- [[电影中的机器意识]]
- [[2026_长片项目_人物设定]]

资源：
- [[AI 与哲学.pdf]]
- [[三峡好人_现实主义笔记]]
- [[神经网络隐喻]]
```

这里同一本书可以进入哲学地图、技术地图、电影项目地图。文件本身不需要移动。

## 树和图的分工

| 场景 | 用树 | 用图 |
|---|---|---|
| 文件备份 | 是 | 否 |
| 批量重命名 | 是 | 否 |
| 媒体刮削 | 是 | 否 |
| 项目交付 | 是 | 辅助 |
| 主题研究 | 辅助 | 是 |
| 灵感连接 | 否 | 是 |
| 跨媒介创作 | 辅助 | 是 |

判断标准：

> 需要机器稳定处理时，用树。需要人类自由联想时，用图。

## 在不同资产中的位置

| 资产 | 树状层 | 块茎/图谱层 |
|---|---|---|
| 电影 | `Movies/片名 (年份)` | 影人、母题、时代、镜头语言 |
| 普通视频 | 项目/课程/来源 | 剪辑灵感、引用关系 |
| 图片 | Photos/Eagle/格式库/项目库 | 分镜、视觉母题、情绪板 |
| 浏览器书签 | 日常入口/工作与经营/技术与AI入口/工具箱 | 网页内容进入 Obsidian 后再关联 |
| Obsidian 笔记 | 功能目录 | 双链、MOC、标签、图谱 |
| 书库 | Calibre 或“系统角色/藏书学科”结构 | 读书笔记、概念地图、跨学科索引 |

## 德勒兹块茎的有用部分

德勒兹（Gilles Deleuze）和加塔利（Felix Guattari）在《千高原》中提出块茎，用来反对单一中心、固定层级、根系式结构。

映射到数字资产管理时，有用的是这些原则：

| 原则 | 文件管理中的可用含义 |
|---|---|
| 任意连接 | 笔记、图片、书、视频可以互相引用 |
| 异质性 | 不同媒介可以进入同一研究语境 |
| 多元性 | 一个对象可以有多个入口 |
| 断裂重组 | 目录不动，索引可以重建 |
| 制图学 | MOC 是地图，不是永久目录 |

但要克制：

- 块茎不是迁移脚本。
- 块茎不是文件命名规则。
- 块茎不能替代媒体库规范。
- 块茎不能帮你决定唯一物理位置。

## 推荐表达

在主技能或汇报中优先使用这些表述：

- “物理层采用单主轴分类。”
- “逻辑层采用标签和 MOC 做多入口索引。”
- “块茎只用于关系发现，不用于物理归位。”
- “这份文件放在唯一位置，但可以出现在多个知识地图中。”

避免这些表述：

- “块茎分类”
- “用图谱替代文件夹”
- “不需要分类，只需要链接”
- “所有文件都扁平化”

扁平化不是目的。真正的目标是：物理层足够稳定，逻辑层足够自由。
