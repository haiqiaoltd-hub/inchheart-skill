# 浏览器书签

浏览器书签是入口，不是知识库。分类主轴不是学科，而是“打开它要完成什么动作”。

核心分工：

> 浏览器保存入口；Obsidian 保存系统。

## 何时留在浏览器

| 类型 | 判断 | 示例 |
|---|---|---|
| 高频入口 | 每周或每天打开 | Google、B站、豆瓣、ChatGPT |
| 工具按钮 | 打开后立即完成动作 | 翻译、识图、视频下载、二维码、格式转换 |
| 账号后台 | 需要登录操作 | 云服务器、DNS、API 控制台、电商后台 |
| 服务购买/检测 | 变化快，临时性强 | API 号商、小作坊中转、VPS、科学上网 |
| 下载入口 | 用完即走 | 磁力、云盘搜索、软件资源站 |
| 当前项目入口 | 只服务正在做的项目 | 项目资料页、协作文档、管理后台 |

## 何时迁移到 Obsidian

| 类型 | 判断 | 建议笔记 |
|---|---|---|
| 知识资料源 | 需要解释、比较、整理路径 | 学科资料索引 |
| 审美资料源 | 能形成图像/音乐/电影参考库 | 艺术摄影与视觉资料源 |
| 长期技术系统 | 影响环境搭建和工具链 | Code 环境搭建全览 |
| 片单/书单/榜单 | 需要排序、注释、关联 | 片单、书单、音乐资料源 |
| 研究网页 | 需要摘录、评论、双链 | 主题研究笔记 |

不要把 Obsidian 变成第二个书签栏。只有当链接背后需要解释、比较、注释、归纳、形成关系时，才迁移进去。

## 推荐浏览器结构

整理后的入口型书签可以使用：

```text
书签栏/
├── 日常入口/
├── 工作与经营/
├── 技术与AI入口/
├── 工具箱/
├── 软件与插件/
├── 影视下载/
├── Github待整理/
└── 临时与灰色入口/
```

### 日常入口

放每天或每周打开的入口。

```text
日常入口/
├── 新闻/
├── 翻译/
├── 识图/
└── 语言学习/
```

规则：

- 高频入口可以直属 `日常入口/`。
- `日常入口/` 不要超过 30-40 个。
- 超过使用频率阈值的链接，降级到其他目录。

### 工作与经营

放事务流和经营入口。

```text
工作与经营/
├── 公司税务/
├── 电商/
├── 自媒体/
├── 房产/
├── 拍卖/
├── 投标/
├── 虚拟储蓄卡/
└── 区块链/
```

规则：

- 后台、查询系统、操作平台留浏览器。
- 具体流程、踩坑、账号说明、策略判断进入 Obsidian。

### 技术与AI入口

放 AI、API、服务器、本地模型、网络代理等操作入口。

```text
技术与AI入口/
├── AI对话与常用入口/
├── API提供商/
├── API中转与检测/
├── API账号与购买/
├── 服务器与域名/
├── 本地模型入口/
├── 技术论坛与榜单/
└── 网络代理/
```

规则：

- API 控制台、状态页、价格页、服务入口留浏览器。
- Code CLI、MCP、Skills、记忆系统、环境搭建方法进 Obsidian。
- 论坛单帖只有验证有效、值得长期引用后才迁移到 Obsidian。

### 工具箱

放一次性工具入口。

```text
工具箱/
├── 综合工具箱/
├── 字体图标/
├── 网页工具/
└── 视频下载工具/
```

规则：

- 工具箱不负责知识分类。
- 不把“前端设计参考”“艺术灵感”放工具箱；这类进 Obsidian 或视觉资料库。

### 软件与插件

```text
软件与插件/
├── 浏览器插件/
└── 软件资源/
```

规则：

- 下载站、插件商店、软件资源站留浏览器。
- 软件评测、安装流程、配置说明进 Obsidian。

### 影视下载

```text
影视下载/
├── 云盘搜索/
└── 磁力搜索/
```

规则：

- 下载入口留浏览器。
- 片单、导演研究、影像资料索引进 Obsidian。
- 不把电影收藏目录和下载入口混在一起。

### Github待整理

用于还没判断长期价值的 GitHub/开源链接。

规则：

- 定期筛选：工具入口留浏览器，长期技术系统进 Obsidian。
- 不要用泛泛的 `收藏/` 做顶层目录；所有书签天然都是收藏。

### 临时与灰色入口

放不适合进入正式资料体系、但暂时还要保留的入口。

规则：

- 命名要诚实，不要混进正式“影视”“学习”“技术”目录。
- 定期清理。

## HTML 导出文件处理

用户常用 Chrome HTML 书签导出文件，例如：

```text
/Users/mac/Desktop/bookmarks_2026_6_8.html
```

处理流程：

1. 先解析 HTML，统计书签数和文件夹数。
2. 判断哪些目录应迁移到 Obsidian。
3. 先写入 Obsidian 笔记并核对链接数量。
4. 备份 HTML 文件。
5. 从 HTML 中删除已迁移目录或链接。
6. 对剩余书签按入口用途重排。
7. 再次解析 HTML，确认书签数量符合预期。

注意：

- 修改导出 HTML 不会直接修改 Chrome。
- Chrome 导入 HTML 通常是追加，不是替换。
- 如果要替换现有 Chrome 书签，用户需要先在 Chrome 书签管理器删除旧目录，再导入清理版。

## 验证

解析导出文件后，检查：

- 书签总数是否符合预期。
- 已迁移目录是否已经从 HTML 中消失。
- 下载、后台、工具入口是否仍保留。
- 顶层目录是否是入口用途，而不是学科主题。
- 是否还存在 `收藏/`、`重要/`、`以后看/` 这类无区分能力的目录。
